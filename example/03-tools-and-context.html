Seam 03 · The tool boundary

Tools and context.

A tool is the unit of work the agent invokes: run(input, ctx) returns a ToolResult payload, and that is the whole of it — pure. The agent context ctx is a read-only ambient input — the pre-staged request, written once by the boundary and never written back to. A domain tool and a UI tool have the identical shape: everything is a tool, even the UI.

▸

The seam at a glance

The eight slots, the same shape as every other seam. Read these, then the deep-dive below.

The seam
A tool — the unit of work the agent invokes. Its job: run(input, ctx) turns the agent's intent into a ToolResult payload, a pure function of input and a read of ctx.
Each side
Inner: the ctx — a read-only staged slice of observation/state — plus the agent's intent expressed as input. Outer: the boundary that folds the returned payload. Same on both: a domain tool (setPriority) and a UI tool (focusTicket) are the same pure shape.
What crosses
In: the input the agent supplies, plus a read of ctx. Out: a ToolResult DTO, returned. Forbidden: a tool that mutates ctx, returns a live handle / entity / socket, or performs an effect. None of those may cross.
Direction
Source dependency points inward — a tool depends only on ctx's read-only shape and on the result types; ctx names nothing about the tool. Results flow outward as a returned payload — never written back into ctx.
The inversion
ctx is written-once-by-the-boundary, read-only-to-tools; the result leaves via the return value and is folded at the boundary. The agent reaches state through data it returns, never through a reference to a store it holds and mutates.
Why it exists
So a tool is tested by a direct call — no model, no surface, no mock of a mutable context. And so adding a capability is one tool plus one fold arm: a closed, local edit, not a thread pulled through the system.
What breaks
A tool that writes its result into ctx (or a shared store) instead of returning it: the fold never sees it, the bus never records it, and re-fold diverges — Hidden State Coupling. replayTest dies.
The test
setPriority.run({4118, High}, ctx) == PrioritySet(4118, High) — by direct call, no doubles. The result depends only on input; ctx is read, never written.
A

What it is

The seam is a tool: the unit of work the agent invokes. The model emits an intent — “set ticket 4118 to High” — as a structured input, and the seam answers with a payload. The whole of a tool is one pure function, run(input, ctx), that takes the agent's input and a read-only context and returns a ToolResult. It reads; it computes; it returns. It does not mutate, it does not reach out, it does not record:

a tool — verbatim from the assembled programpseudocode
// A DOMAIN TOOL — pure. Reads read-only ctx, returns a payload, mutates nothing.
DOMAIN_TOOL setPriority {
  input = { ticket: TicketId, level: Priority }
  run(input, ctx) -> PrioritySet { ticket: input.ticket, level: input.level }
}
// A UI TOOL — also pure, also returns a payload. Drives the surface, decides nothing.
UI_TOOL focusTicket {
  input = { ticket: TicketId }
  run(input, ctx) -> Focused { ticket: input.ticket, known: ctx.tickets.has(input.ticket) }
}

Two tools, one shape. setPriority derives its payload purely from input. focusTicket reaches a little further — it reads ctx.tickets to set known — but the read is the only contact it has with the world, and it is a read, not a write. Both return a ToolResult and stop. The payloads they may return are the closed set of result types:

tool results — the DTOs a tool may returnpseudocode
sealed ToolResult {
  PrioritySet(ticket: TicketId, level: Priority)
  EscalationRequested(ticket: TicketId, by: Actor)
  EscalationConfirmed(ticket: TicketId, by: Actor)
  Focused(ticket: TicketId, known: Bool)
}

The agent context — ctx — is the other half of the seam, and the more easily mistaken one. It is a read-only ambient input: a staged slice of observation and state, written once by the boundary before the step runs, and exposed to the tool as immutable. It is the pre-staged Request Model: in Clean Architecture the controller hands a Request Model across the Input Port at call time; here the boundary has already deposited everything the tool needs as ambient data before the step, so the tool merely reads it. The Input Port proper is the tool's run signature — the inner-side seam the agent invokes; ctx is the request data, staged rather than passed across at call time.

B

A UI action is a tool too — the symmetry

The headline of this architecture is that everything is a tool, even the UI, and this seam is where that claim is either true or hollow. It is true here, exactly, because setPriority and focusTicket have the same pure shape. One mutates domain meaning (a ticket's priority); the other drives the surface (which row is focused). Yet neither performs the change. Both compute a ToolResult and return it; both leave the actual change — updating the model, or marking the focused row — to the fold and the projection downstream.

setPriority · a domain tool

Input is the agent's intent over the domain. run returns PrioritySet(ticket, level) — a fact, not a write. The fold turns that fact into a new State and a Log effect.

focusTicket · a UI tool

Input is the agent's intent over the surface. run reads ctx.tickets for known and returns Focused(ticket, known) — also a fact, not a write. The same fold path carries it; the projection reflects it.

Because the shape is identical, a UI action and a domain action are peers: each is a tool, each returns a ToolResult, each is folded the same way and recorded on the same bus. There is no second code path for “the UI doing something” — the surface emits an intent, the agent (or the host's tap) invokes a tool, and the payload joins the one stream. This is why the slogan is a data model and not a metaphor: the UI tool is not like a domain tool, it is one. Map this to Clean Architecture and each is invoked across the same kind of Input Port — the run seam a caller crosses to reach a use case — the only difference being which kind of intent the caller is expressing.

The same DTO law as seam 02

What crosses out of a tool is a ToolResult — plain data, a member of a closed sealed set. Never an entity, never a live Ticket object the boundary could mutate, never a socket or a file handle. The tool answers the question “what happened?” with a value, and hands that value back. Everything that acts on the value happens later, at one seam, where it can be recorded and replayed.

C

Why the context is read-only, and the result returned

The discipline has two halves and they are the same idea seen from both ends. Inbound: ctx is written once by the boundary and is read-only to tools. Outbound: the result leaves by the return value and is folded at the boundary. Put together, a tool has exactly one channel in (read ctx + take input) and exactly one channel out (return a payload). No hidden side channel exists, because none is reachable: the tool was never handed a mutable reference to anything.

This is the inversion. Naively, a tool that needs to “set priority” would hold the store and write to it. Here it cannot — it is given a read-only ctx and asked to describe the outcome as data. The capability to actually change state is not in the tool's hands; it is in the fold's, reached only through the value the tool returns. Control flows outward (the boundary acts on the result), but the source dependency points inward (the tool depends on result types and on ctx's shape, and on nothing outer).

The payoff is two properties stated plainly. Testing collapses to a function call: with no mutable context to mock and no effects to intercept, you assert run(input, ctx) == expected and you are done — no model in the loop, no surface, no doubles. Extension is local: a new capability is one new tool plus one new arm in the fold's match. You do not thread a new method through a store, a controller, and a presenter; you add a result type, a tool that returns it, and a fold arm that consumes it.

The Input Port, and the pre-staged request

In Clean Architecture the Input Port is the interface the controller calls to pass the Request Model in. The example keeps that split but stages it: the tool's run signature is the Input Port — the inner-side seam the agent invokes — and ctx is the Request Model, deposited as ambient data before the step rather than handed across at call time. That staging is exactly what makes a tool a pure function: there is nothing to call back through, only data to read and a value to return.

D

What breaks without it

The seam earns its keep only against the version that omits it. Here is the most tempting violation — a tool that writes its result where it stands, into the context, instead of returning it:

Collapsed — the tool writes its result into ctx

DOMAIN_TOOL setPriority {
  run(input, ctx) -> { ctx.tickets[input.ticket].priority = input.level }  // mutates ctx, returns nothing
}

It looks shorter — but ctx is read-only (the seam's one rule), so the write lands on the staged copy the boundary discards when the step ends: the priority never actually changes, and nothing is recorded. The test that dies: replayTest. The fold receives an empty step.toolResults, so it produces no new state and no Log effect; the boundary appends nothing meaningful to the bus, so the append-only log no longer contains the priority change. Re-folding the recorded stream yields a different state than the live run had — the “fold twice, assert identical” check fails, and worse, replay silently diverges from reality. This is Hidden State Coupling: the result took a side channel the recorder cannot see. You did not save a line; you deleted replay.

Collapsed — the tool returns a live handle

If run returns the live Ticket object (or a socket, or a file handle) instead of a ToolResult value, the boundary now holds something it can mutate and the world can change underneath. The payload is no longer a record of what happened; it is a leak of where it happened. Replay cannot reconstruct it — a handle is not data — and two folds of the same stream can disagree because the thing behind the handle moved. The DTO rule exists to make this impossible: a tool may return only a closed, inert value.

Both failures share a shape: the tool tried to act — to write, or to hand out a live thing — instead of describing the outcome as data it returns. The seam exists precisely to make that the only move available, so the recorder sees everything and replay never diverges.

E

The test that the seam makes possible

Because a tool is a pure function of input and a read of ctx, it is tested by direct call — no model, no surface, no mock of a mutable context, no doubles at all:

testing a pure tool — a direct callpseudocode
test "setPriority is a pure function of its input":
  ctx = stagedContext(tickets = { 4118: ticket(4118) })
  assert setPriority.run({ ticket: 4118, level: High }, ctx)
         == PrioritySet(ticket: 4118, level: High)        // returned, not written

test "focusTicket reads ctx, mutates nothing":
  ctx = stagedContext(tickets = { 4118: ticket(4118) })
  assert focusTicket.run({ ticket: 4118 }, ctx) == Focused(ticket: 4118, known: true)
  assert focusTicket.run({ ticket: 9001 }, ctx) == Focused(ticket: 9001, known: false)
  assert ctx.tickets == { 4118: ticket(4118) }            // ctx unchanged — read-only held

The first test is the canonical one: setPriority.run({4118, High}, ctx) == PrioritySet(4118, High), a plain equality on values. The focusTicket case shows the symmetry — the UI tool is tested identically, reading ctx to compute known and asserting that ctx came out unchanged. No model emits the call; no boundary commits anything; no effect fires. That is the whole gift of the seam: the unit of work the agent invokes is a value-in, value-out function, so the test is an equation. The impure work — recording the payload, folding it into state, performing effects — lives one seam outward, at the boundary adapter, which is the one object tested with doubles. Everything here is pure, and pure things are tested by asking them a question and checking the answer.
