Seam 05 · Ports and adapters

Ports and the swap.

Three interfaces the core owns and adapters implement outward — Runtime, ModelProvider, EventSource. The port is declared inside, in the core's own vocabulary; the adapter is bound outside. That single inversion buys the headline move: the same core runs behind a live adapter or a fake one, so the whole session replays deterministically off-target. A test case is the application's first user.

▸

The seam at a glance

The eight slots, the same shape as every other seam. Read these, then the deep-dive below.

The seam
A port — an interface stated in the core's own vocabulary. Runtime (dispatch a Command, observe a Stream<State>), ModelProvider (the model backend), EventSource (sensing — tickets in). The core declares them; adapters satisfy them.
Each side
Inner: the core declaring the port — a naked interface, no implementation. Outer: an adapter implementing it — HttpModel talking to a backend, TicketQueue draining a queue, or their fakes. One port, many adapters.
What crosses
In: a Command via dispatch, a RawEvent via EventSource. Out: a Stream<State> via observe, a StepResult via ModelProvider. Forbidden: a live socket, a DB row, or a model handle crossing inward — only domain types pass the port.
Direction
Source dependency points inward: the adapter depends on the core's port interface (DIP) — never the reverse. Control flows outward at runtime — calls travel out through the port to the backend and results return.
The inversion
The port is declared inner, implemented outer. The compile-time arrow (adapter → port) opposes the runtime arrow (core → backend). The polymorphic interface on the inner side is the hinge that lets the two run in opposite directions.
Why it exists
Substitutability. Because the core names only the port, any adapter that fits it is interchangeable — and a fake one makes the whole session replay off-target, with no backend, no socket, no clock of its own.
What breaks
Let the core call HTTP or SQL directly with no port and it can no longer run without the backend and can no longer replay. Collapse the live and fake adapters into one wired pair and replayTest becomes impossible to write.
The swap
wireApp(env.live = false) binds ScriptedModel in place of HttpModel — same core, deterministic, off-target. Tests drive the application through the port as its first user; nothing in the core notices the swap.
A

What it is — three interfaces the core owns

A port is not a class and not an adapter. It is an interface, declared in the inner ring, stated entirely in domain words — and nothing more. The core owns three of them, and each names a conversation the core needs to have with the world without ever naming how that conversation happens:

the ports — verbatim from the assembled programpseudocode
PORT Runtime       { dispatch(Command); observe() -> Stream<State> }   // the domain port
PORT ModelProvider { step(prompt, tools) -> StepResult }              // the model backend
PORT EventSource   { events() -> Stream<RawEvent> }                    // sensing (tickets in)

Read what is — and is not — in those three lines. Runtime takes a Command in and hands a Stream<State> out: it is the domain port, the seam the surface and the sensing loop both push against. ModelProvider takes a prompt and tools and returns a StepResult — the model backend, reduced to one verb. EventSource yields a Stream<RawEvent> — sensing, the queue of tickets arriving. No URL. No socket. No SQL. No model handle. Each port speaks only the core's vocabulary; the protocol on the far side is the adapter's private business.

That is the discipline that makes the seam a seam: the interface lives inside, in the core's language, and the implementation lives outside. Clean Architecture calls these the gateway / repository interfaces and the Input / Output Ports; the mapping is close. dispatch plays the Input Port (the world calls in), observe the Output Port (the core pushes out), and ModelProvider / EventSource are gateway interfaces the core declares and adapters satisfy — though the reference bundles these into one DomainPort — whose signature is onAction(Action) plus an observed state: Observable<State>, which this example renames Runtime.dispatch / observe for the running console (the Action is what crosses the port; the boundary signs it into the Command). Read this as Clean Architecture's gloss on a single seam.

One port, many adapters

The same port interface admits more than one implementation, and that plurality is the seam's value. On the live side, an adapter talks to the real world; on the fake side, an adapter replays a fixture. Both fit the same port, so the core cannot tell them apart — and neither can replayTest.

B

What crosses it — domain types only, both ways

A port is a promise about what crosses, and the promise is narrow: only domain types. Inbound, a Command arrives through dispatch and a RawEvent arrives through EventSource.events. Outbound, a Stream<State> leaves through observe and a StepResult leaves through ModelProvider.step. Every one of those is plain data the core already understands. What may not cross is the whole reason the port exists:

Forbidden inbound — a live handle masquerading as a domain type

A live socket, an open DB row cursor, a raw model session handle — none of these may cross the port inward. The moment one does, the core holds a reference to a mechanism, the source dependency flips outward, and the fake adapter can no longer stand in for the live one because the core is reaching through the port at something only the live side has. The port stops being substitutable, which is the only property it was there to provide.

So the rule is symmetric with every other seam on this site: data crosses, mechanism does not. The StepResult that comes back from ModelProvider.step is a value — tool calls and their ToolResult payloads — not a live model object. The RawEvent from EventSource is a value normalized by the adapter, not the network packet itself. The port is the membrane that keeps the core's vocabulary clean of the backend's, in both directions, so either side can be replaced without the other noticing.

C

Why it exists — the inversion, and substitutability

The port earns its place through one move, the same move Clean Architecture builds its outer rings on: the polymorphic interface is declared on the inner side and implemented on the outer side. The source-code arrow runs adapter → port (the adapter names the core), while the runtime arrow runs core → backend (control flows out through the port to the model or the queue). The two oppose each other, and the interface is the hinge that allows it.

That opposition is exactly the “dependency direction vs. flow of control” principle stated in the overview's law, made concrete at this seam. Because the dependency points inward, the core never knows which adapter it is talking to. Because control flows outward, the work still gets done. And because the only thing the core holds is the port, any adapter that fits the port is a legal substitute — including a fake one that touches no backend at all.

Substitutability is not an abstract virtue here; it is the load-bearing property. The reason to invert the dependency is precisely so that the live adapter and the fake adapter are interchangeable behind one port. With that in hand, the whole session can be re-run off-target — no socket, no database, no model server — by binding the fake. The port is what makes “run the real session, deterministically, with nothing real attached” a thing you can actually do.

The Dependency Inversion Principle, in one seam

High-level policy (the fold) must not depend on low-level detail (an HTTP client). Both depend on the abstraction — the port — and the port belongs to the policy, in the policy's vocabulary. That is DIP verbatim, and it is the difference between a core that can only run against production and a core that can run against a recording.

D

What breaks without it

The seam earns its keep against the version that omits it. The most tempting shortcut is to skip the port entirely and let the core reach for the backend directly — it is fewer lines and it works on the happy path:

Collapsed — the core calls the backend directly, no port

fold(state, results, now):
  result = HttpModel("https://api/…").step(prompt)  // reaches the socket itself
  row   = db.query("select * from tickets")     // and the database
  …

Now the source dependency points outward: the core names HttpModel and db. The core can no longer run without the backend — every test needs a live socket and a live database — and there is no port to bind a fake against, so the session can no longer replay off-target.

There is a subtler collapse that keeps a port-shaped interface but defeats its purpose: wiring the live and fake adapters together into one inseparable pair, so they always ship as a unit and can never be swapped independently.

Collapsed — live and fake fused into one wired pair

model = HttpModelWithScriptedFallback(env.url, env.fixtures)  // two adapters welded together
  // step() hits the network, then "verifies" against the fixture in the same call

This looks defensive, but it removes the swap. There is no env.live = false path that binds only the fake — every run still touches the network, because the fake is soldered to the live adapter rather than standing in for it. The test that dies: replayTest. It can no longer run, because there is no way to wire the core to fixtures alone — collectPerform(recorded, mode = REPLAY) still reaches the backend, so “fold the recording, assert the golden effects” can never be performed off-target. The two-adapter weld deletes deterministic replay.

Both failures share a shape with every other seam on this site: a capability that belongs to the adapter — knowing the URL, holding the socket, choosing the backend — leaked into the core, where it can be neither swapped nor replayed. The port exists precisely to make that leak impossible to write by accident, and to keep one wire, the composition root's, the only place a live-vs-fake choice is ever made.

E

The swap — a test is the application's first user

Here is the payoff the seam was built for. The composition root binds exactly one adapter per port, and for ModelProvider the choice is a single ternary — the swap line, lifted verbatim from the assembled program:

the composition root — the one place the swap is decided (seam 06)pseudocode
wireApp(env):
  bus     = Bus()
  store   = LogStore(env.path)
  model   = env.live ? HttpModel(env.url) : ScriptedModel(env.fixtures)   // ← the swap
  sensing = TicketQueue(env.queue)
  runtime = Runtime(model, bus, store, boundary)
  return App(runtime, sensing)

One line, two worlds. With env.live = true the core runs behind HttpModel against a real backend. With env.live = false the same core runs behind ScriptedModel reading recorded fixtures — deterministic, off-target, no socket. Put the two wirings side by side and the symmetry is the whole lesson: only the adapter changes; the runtime, the fold, the bus, the ports are byte-for-byte identical.

live vs. fake — same core, one bound argument apartpseudocode
// live: drive the real backend through the port
liveModel = HttpModel(env.url)
liveApp   = App(Runtime(liveModel, bus, store, boundary), TicketQueue(env.queue))

// fake: drive a recording through the SAME port — the swap, made explicit
fakeModel = ScriptedModel(env.fixtures)
fakeApp   = App(Runtime(fakeModel, bus, store, boundary), TicketQueue(env.queue))

// the core cannot tell liveApp from fakeApp — it holds only ModelProvider
assert liveApp.runtime.port == fakeApp.runtime.port    // both fit PORT ModelProvider

This is where the borrowed idea lands hardest. In Clean Architecture, the outermost ring is the tests, and a test case is treated as the first user of the application — it drives the system through the very same ports any other client would, substituting a fake for whatever the system would otherwise reach out to. The swap above is that idea, mechanized: bind ScriptedModel and the test harness becomes a driving adapter on one side while recorded fixtures become a driven adapter on the other, and the real session runs between them with nothing real attached.

And so the seam delivers what every other page has pointed at: the replay harness can fold the recorded stream twice and assert determinism, because the fake adapter lets it run off-target. The whole session — sensing, stepping, folding, projecting — replays behind fakes that fit the same ports the live adapters fit. One port, two adapters; the core never notices the difference; the test drives it as its first user. That is the hexagonal swap, and it is the single most important idea this architecture borrows.
