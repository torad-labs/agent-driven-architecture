Seam 07 · The payoff, and more boundaries

Replay and the advanced seams.

The cash value of every seam before it. Because the bus is append-only and the fold is pure, you fold the recorded command stream twice and assert an identical State and identical golden effects — with no model, no network, no clock, no surface. This is the outermost circle: the test that proves every other seam held.

▸

The seam at a glance

The eight slots, the same shape as every other seam — the last time you read them. This is where seams 01–06 get cashed.

The seam
The harness that proves every other seam holds. Its job: re-derive the whole session from the recorded stream alone, twice, and show the two derivations are identical.
Each side
Inner: foldAll over the recorded stream — pure, the same fold the live path runs. Outer: the recorded commands plus the captured fixtures — a file, not a world.
What crosses
In: the recorded log — signed Commands plus captured ToolResult fixtures. Out: a State and a list of Effects, compared to golden. Forbidden: re-hitting a live source on re-fold — a tool result that depends on the world must be a recorded fixture, never recomputed.
Direction
All dependency points inward at the fold; nothing points out to a clock, a model, or a sink. There is no control flowing outward here at all — REPLAY mode performs nothing on the world.
The inversion
The fake port sits behind the domain port: where the live root wired HttpModel and a clock, the harness wires recorded fixtures and a fixed clock. The core cannot tell — it was always handed values, never a world.
Why it exists
Replay is the concrete payoff of purity: debugging is replay, there is no hidden mutable state to reproduce, and a regression is a stored stream that re-folds wrong. Every prior seam exists so this one can.
What breaks
ANY impurity leak from seams 01–06 makes this test impossible. If a tool or the fold read the clock, hit the network, or mutated shared state, folding the same stream twice diverges — and replayTest cannot exist.
The test
foldAll(initialState, recorded) twice; assert the two States are equal (determinism), and assert the replayed effects equal recorded.goldenEffects. No doubles beyond the recording itself. (foldAll is just the step-level fold run over the whole timeline.)
A

What it is

Clean Architecture's last word on the rings is a quiet one: tests are the outermost circle. They depend inward on everything and nothing depends on them, so they are free to be ruthless. This architecture takes that literally and hands you the sharpest possible instance of it. The whole session was committed to an append-only bus at the boundary — signed Commands and their ToolResult payloads, in order, off the record nowhere. That recording is everything you need to re-derive the run. So the test is brutally short:

the replay harness — verbatim from the assembled programpseudocode
replayTest(recorded):                           // recorded = signed commands + fixtures
  a = foldAll(initialState, recorded)
  b = foldAll(initialState, recorded)
  assert a.state == b.state                      // determinism (G9)
  assert collectPerform(recorded, mode = REPLAY) == recorded.goldenEffects

Four lines, and there is no model in them, no network, no clock, no surface. foldAll threads the recorded results through the same pure fold the live path runs — the one that takes now as an ordinary value — starting from a fixed initialState. Run it once, run it again: if seams 01–06 held, the two States are bit-identical, and the effects the run would have performed, collected in REPLAY mode (described, never fired), match the golden list captured the first time. That is the entire payoff of every earlier seam, made executable.

B

What crosses it — recorded data, never a live world

Exactly one thing crosses into the harness: the recorded log. It is plain data — signed Commands and the ToolResult fixtures captured beside them at commit time — and it is the same append(bus, signed, results) record the boundary wrote at line (4) on page 02. Nothing else may cross. The forbidden payload here is not an entity or a live handle; it is subtler and more tempting: re-hitting a live source on re-fold.

The rule is a single sentence. A tool result that depends on the world must be a recorded fixture, not recomputed. If focusTicket returned Focused(ticket, known) by querying a live index the first time, replay must read the captured Focused value back from the stream — it must not re-run the query. The fold never sees the difference, because the fold only ever consumed ToolResult data; whether that data came from a live tool or from the recording is a fact about the outer ring, settled at composition. This is the same inversion as seam 05, one notch deeper: where the root swapped HttpModel for ScriptedModel, replay swaps the entire sensing-and-tool surface for a recording, and the pure core is none the wiser.

Why “commit before perform” is what makes this possible

Recall the boundary's fixed order: commit (4), then perform (6). Because the signed command and its results are appended before any effect fires, the recording is a complete and faithful account of every decision — including ones whose live effects later failed. Replay folds that account and stubs the effects entirely; it does not need them to have happened, only to have been recorded as intended. (Dedup-on-key belongs to recovery, which re-drives real effects after a crash; replay performs nothing, so it has nothing to dedupe.) The append-only bus is not a log you keep for forensics; it is the input to the test.

C

Why replay is the whole point

Most architectures earn testability seam by seam and then test against a thicket of doubles: a fake clock here, a stub gateway there, a recording sink somewhere else, each standing in for a piece of the world the code reaches out to touch. This architecture spent six seams pushing every reach-for-the-world into one auditable place so that the test at the end needs none of that scaffolding. The clock was injected, so a fixed clock reproduces time. Identity was minted at one seam, so the recorded ids are the ids. Effects are descriptors, so REPLAY mode collects them without firing them. The fold is pure, so it is a mathematical function of (initialState, recorded) — and a function called twice on the same input returns the same output, by definition.

That is why the payoff is stated as debugging is replay. A production incident is not a mystery to be reconstructed from logs and intuition; it is a recorded stream you fold on your laptop, deterministically, until the line that misbehaves reveals itself. A regression test is not a hand-built fixture; it is yesterday's stream, asserted to still re-fold to the same State. There is no hidden mutable state to reproduce, because the architecture forbade hidden mutable state at every seam that came before. Replay is not a feature bolted on at the end — it is the shadow that purity casts.

D

What breaks without it

This seam is unusual: it has no “collapsed version of itself” to show, because the failure is never here. The harness is four pure lines; there is nothing in it to get wrong. What breaks replayTest is an impurity leaked into any earlier seam — and this page is where that leak finally gets cashed as a failing assertion. Here is the most common one, the same temptation seam 02 warned against, now viewed from the test that dies for it:

Collapsed — a tool reads the world instead of returning a fixture

UI_TOOL focusTicket {
  run(input, ctx) -> Focused { ticket: input.ticket,
    known: liveIndex.query(input.ticket) }  // reads the world, not ctx
}

It looks harmless — it just answers truthfully from the freshest source. But known is now a function of the live index, not of recorded data. The test that dies: replayTest. On re-fold, foldAll re-runs the tool against a liveIndex that has since changed (or is simply absent off-target), so fold number two produces a different Focused than fold number one — a.state == b.state fails. Even with a stable index, the harness can no longer run with no network, which was the whole promise. You have not made the tool more accurate; you have deleted replay.

The shape generalizes to every prior seam. If the fold read the clock instead of taking now as a value, the two folds time-stamp differently and a.state == b.state fails. If the boundary stamped the Actor from a live source rather than recording it, replay reconstructs the wrong authorship. If two folds were allowed to interleave (see § E below), State is corrupted before it is ever compared. Every “what breaks” on every earlier page was, in the end, a sentence about this test. That is why it is the outermost circle: it is the single assertion that every inner seam was telling the truth.

E

More boundaries — advanced, optional

Three further seams, each an opt-in rung, not part of the core. The seven seams you have read are the architecture; these are what you add when the load demands it. Each gets the same four-line treatment — and each, note, is ultimately defended by the same replay test above.

These are deliberately compact. Reach for them when a concrete pressure appears — a slow deep model, bursty input, generated code at volume — and not before. None of them changes the rings; each one earns its place by protecting the very purity that replayTest depends on.

Seam · the tiered relay
A slow deep tier and the fast tier communicate only through an append-only relay store, recalled by the fast tier through one bounded tool. What crosses: recorded relay entries — data, read back, never a shared handle. Why: two cognitive speeds without one stalling the other, and the relay is itself replayable. What breaks: shared mutable state between the tiers kills replay — the deep tier mutating the fast tier's State directly is exactly the interleaving the harness can no longer reproduce.
Seam · the barge-in mailbox
A serial consumer with newest-input-wins conflation and interrupt-preemption (cancel-and-join): a new input cancels the in-flight turn and joins on it before starting, so two folds never overlap. What crosses: one staged event at a time into a read-only ctx. Why: bursty or interrupting input without dropping the single-writer guarantee. What breaks: concurrent folds corrupt State — without cancel-and-join, two turns write the same State and a.state == b.state stops being meaningful.
Seam · the enforcement gate
The invariants checked as blocking build gates: actor-stamped at the boundary, tool purity, dependencies point inward, irreversible action gated behind a confirm. What crosses: nothing at runtime — it is a compile-time fence around the seams. Why: the other seams must survive generated code at volume, where a leak is one careless line away. What breaks: without the gate the seams erode silently — an ungated perform or a forged Actor ships, and the first symptom is replayTest going red weeks later.

Read those once for awareness, not mastery. The point is the asymmetry: the core seven seams are load-bearing and you cannot skip them; these three are rungs you climb only when a real pressure asks for them. And every one of them is, in the end, in service of the four pure lines at the top of this page — the test that says the whole session re-derives, deterministically, from a recording alone.

That closes the worked example. One command, traced through every seam; one impure boundary; one law — source dependencies point inward; and one test that cashes them all. Return to the overview to walk the seams again, or open the full architecture reference the example was built to teach.
