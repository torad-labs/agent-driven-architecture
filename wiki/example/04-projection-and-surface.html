Seam 04 · The humble object

Projection and surface.

The pure formatter that decides every presentational flag, and the dumb shell that only moves fields to widgets. This is the Humble Object split — Clean Architecture's Presenter / View / View-Model, point for point. All the logic on the side you can inspect; an empty shell on the side that drifts.

▸

The seam at a glance

The eight slots, the same shape as every other seam. Read these, then the deep-dive below.

The seam
The split between project — pure, pre-computing every presentational flag — and the render surface, which applies those flags and decides nothing. One half holds all the logic; the other holds none.
Each side
Inner: State — the truth. Middle: project — the testable formatter that turns truth into primitives. Outer: render — the dumb shell that moves a TicketRow's fields onto widgets.
What crosses
In: the State. Out: a ConsoleView of TicketRows — formatted primitives only: badge text, canEscalate bool, escalating bool. Forbidden: the surface computing a flag (if t.status == Open) itself; project doing any I/O.
Direction
project depends on State only; the surface depends on the ConsoleView only. Neither imports the fold or an adapter — the formatter knows nothing of the screen, the screen knows nothing of the truth.
The inversion
Every decision is pre-computed in project; render only applies it. The View-Model is a polymorphic plain-data buffer between them — the surface reads it and obeys, never reaches behind it to the State.
Why it exists
To put all presentational logic on the side you can inspect by value (the View-Model) and leave an empty shell on the volatile side. The view becomes trivially testable and holds nothing that can drift.
What breaks
Let the surface decide canEscalate itself and you mint a second source of truth that drifts from State — and you cannot test it without rendering the screen. render() with an if over t.status is the collapse.
The test
A pure call, no screen: project(stateWithHigh).rows[0].badge == "HIGH", and .canEscalate == false once the ticket is Escalating. Inspect the View-Model; never render to assert.
A

What it is — a Presenter and a humble View

The seam carries one conversation: how does the current State become pixels? It splits that conversation in two. On the testable side, project reads the State and returns a ConsoleView — a bag of formatted primitives in which every presentational decision has already been made. On the humble side, render takes that view and walks it, moving each field onto a widget. Here is both halves, verbatim from the assembled program:

the view projection — the Presenter, purepseudocode
project(state) -> ConsoleView:
  ConsoleView {
    rows: state.tickets.values.map(t -> TicketRow {
      ticket:      t.id,
      badge:       badgeFor(t.priority),        // "HIGH" + colour, pre-computed
      canEscalate: t.status == Open,            // the flag is decided HERE, not in the view
      escalating:  t.status is Escalating }),
    banner: bannerFor(state.run)
  }
the surface — the View, humblepseudocode
render(view: ConsoleView):
  for row in view.rows: drawRow(row)            // no `if` over data, no computed flag
onAction(intent) -> dispatch(intent)            // one sink; the boundary signs it

Look at what each side touches. project evaluates t.status == Open, runs badgeFor, asks t.status is Escalating — it makes all the calls a human eye would call “deciding how this looks.” render makes none: it iterates rows and draws. There is no if over the data in render, no comparison, no derived boolean — only drawRow. The whole presentational policy lives in the first block; the second block is a courier.

This is Clean Architecture's Humble Object pattern, point for point. That book's argument runs like this, in our own words: behaviours that are hard to test get separated from behaviours that are easy to test, with a thin humble object holding the hard-to-test part and as little logic as can be managed. For a screen, what is hard to test is the act of painting widgets; what is easy to test is computing the strings and flags that the painting will use. So you cleave the painting in a humble View that does nothing but copy fields, and you move the computation into a Presenter that produces a View-Model of plain primitives. Our render is that humble View; our project is that Presenter; our ConsoleView / TicketRow is that View-Model. (See Clean Architecture, Ch. 23, “Presenters and Humble Objects.”) The mapping is not an analogy here — it is the same three objects.

The Humble Object rule

The View-Model contains nothing but formatted primitives, and the surface contains no logic that produces them. Every string, every flag, every boolean a widget needs is computed in project and placed in the ConsoleView by value. render may read those fields and place them; it may not derive one. If a decision is being made on the surface, the split has already failed.

B

What crosses it — formatted primitives, never the entity

One data structure crosses this seam, and it is plain data: a ConsoleView holding a list of TicketRow. What it carries is the entire point — it carries conclusions, not the material to draw them from:

Field on TicketRowTypeWhat it already decided
ticketTicketIdwhich row this is — an opaque id, not the Ticket entity
badgeTextthe literal label, e.g. "HIGH" — badgeFor(priority) already ran
canEscalateBoolwhether the escalate control is live — t.status == Open, already evaluated
escalatingBoolwhether to show the in-flight affordance — t.status is Escalating, already evaluated

What does not cross is as load-bearing as what does. The TicketRow never carries the Ticket entity, never the TicketStatus union, never the State. If the raw status crossed, the surface would be able to ask if row.status == Open — and the temptation would, eventually, be taken. By handing across canEscalate: Bool instead of status: TicketStatus, the projection removes the surface's capacity to decide. The View-Model is deliberately impoverished: it gives the surface exactly enough to paint and not one field more.

And the forbidden direction on the other side: project does no I/O. It reads a State value and returns a ConsoleView value. It does not read a clock, open a socket, touch the bus, or perform an Effect. Time and identity belong to the boundary, which already read them before the fold ran; by the time project sees the State, everything in it is settled data. That is why project is a pure function of one argument — and therefore testable by direct call.

C

The direction — and the one sink back

Two dependencies, both pointing the same way, neither reaching past its neighbour. project imports the State and nothing else — it does not know a screen exists, does not name render, does not name a widget toolkit. The surface imports the ConsoleView and nothing else — it does not know the State exists, does not name the fold, does not name an adapter. Source dependencies point inward at the View-Model and the truth; the screen sits furthest out and is named by no one inner.

The split has a second edge that is easy to miss: the surface emits exactly one thing back, and it emits it without deciding anything. onAction(intent) -> dispatch(intent) is a single sink. A tap does not run a handler that branches on state; it hands an intent to dispatch, and the boundary is what signs it into a Command with the Actor stamp. So the surface decides nothing on the way in (it draws pre-computed flags) and nothing on the way out (it forwards one intent). It is humble in both directions.

That is what makes a human tap and an agent tool-call peers at this seam too. The surface that a host taps emits the same intent the agent's path would; the boundary stamps the only difference. The View is not a special “human” entry point with its own logic — it is a courier that happens to be driven by fingers.

D

Why the logic sits on the testable side

The reason to split here is not tidiness; it is testability and drift. State the property the split buys you in one sentence: all presentational logic lives where you can inspect it by value, and the volatile half holds nothing that can be wrong.

Consider the two halves as test subjects. project is a pure function from State to ConsoleView; you call it with a constructed state and read the returned struct. No screen, no toolkit, no event loop — a value in, a value out, asserted by equality. render, by contrast, is the part that genuinely needs a screen to exercise, and is therefore expensive and flaky to test. The Humble Object move is to make that expensive-to-test object hold so little that there is nothing in it worth testing: it copies fields, and copying fields cannot be subtly wrong in a way a presentational test would catch. You concentrate the logic on the cheap side and starve the costly side of logic.

Drift is the second payoff, and it follows from there being one source of truth. Every flag the screen shows is computed once, in project, from the State. There is no parallel computation in the surface that could fall out of step. When the rule for canEscalate changes — say it becomes t.status == Open and t.priority >= High — you change one line in project, and every surface that renders a TicketRow is correct by construction, because none of them ever decided canEscalate for themselves. The view holds nothing that can drift because the view holds no policy at all.

A View-Model is data, not a viewer onto the State

The ConsoleView is a snapshot of conclusions, computed at one instant from one State. It is not a live window the surface peeks through to ask the State questions. That distinction is the whole guarantee: because the surface holds values and not a reference back to the truth, it cannot re-derive a flag, and so it cannot disagree with the projection that produced it.

E

What breaks without it

The seam earns its keep against the version that omits it. The tempting violation is to skip the View-Model and let the surface read the entity and decide for itself — it looks shorter, and on the happy path it works:

Collapsed — the surface decides the flag itself

render(tickets):
  for t in tickets:
    drawRow(t)
    if t.status == Open: drawEscalateButton(t)  // a flag, decided on the surface

Now the rule “you may escalate an Open ticket” lives in two places: in project (if it still exists) and here, inside render. That is a second source of truth, and the two will drift the first time the rule changes in one and not the other — the projection says canEscalate: false for an Escalating ticket while the surface, reading only status == Open, quietly does its own thing. The test that dies: project(stateWithHigh).rows[0].canEscalate == false still passes against the projection — and tells you nothing, because the surface is no longer obeying it. To check the actual behaviour you now have to render the screen and inspect a drawn button, which is exactly the expensive, flaky test the Humble Object split existed to delete. You did not save a struct; you moved a decision somewhere it can neither be inspected by value nor trusted to match the truth.

The shape of the failure is the same one every seam guards against: a responsibility that belongs on the inner, testable side leaked to the outer, volatile side, where it can be neither audited by value nor kept in step. The projection exists precisely so that the only honest way to know what the screen will show is to read a ConsoleView — never to render one.
